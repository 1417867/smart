### 1.2 AI 驱动系统的测试挑战

AI 系统测试面临传统测试框架无法覆盖的独特挑战：

| 挑战 | 描述 | SmartLearn 应对策略 |
|------|------|-------------------|
| **非确定性输出** | LLM 每次生成结果不同 | 不测试具体内容，测试**结构约束**和**类型安全** |
| **外部依赖脆弱** | AI 提供商 API 可能故障 | Mock 外部调用 + 降级路径验证 |
| **流式数据复杂** | SSE 17 种事件类型需要完整验证 | 逐事件类型断言 + 序列号单调性 |
| **Token 消耗巨大** | 每次 LLM 调用都有成本 | 单元测试使用 fallback 模式，E2E 使用 mock |
| **上下文窗口管理** | 长对话需要自动裁剪 | 验证 >90% 阈值触发 snip 逻辑 |
| **向量检索精度** | pgvector 余弦距离需要高精度 | roundtrip 验证 + 固定向量测试 |
| **工具调用链路长** | LLM → Tool → LLM 多轮 | AgentLoop 轮次上限验证（max 20） |
| **记忆一致性** | L1/L2/L3 三层需保持同步 | 跨层读写一致性测试 |

### 6.5 SSE 流事件完整性验证

系统有 **17 种 SSE 事件类型**，流完整性验证确保：

```
StreamEvent 类型          验证点
─────────────────────────────────────────────
stage_start / stage_end   成对出现，stage 名称匹配
thinking                  thinking block 正确解析
content                   文本增量无重复/丢失
tool_call                 参数 schema 与工具定义一致
tool_result               结果与调用的 tool_call_id 对应
progress                  进度百分比单调递增
sources                   引用格式 [N] Title (URL): snippet
error                     错误消息 + 上下文信息
session / session_meta    会话 ID + turn ID 一致
done                      必定是最后一条事件
wait_for_input            交互式等待, 非流结束
session_complete          会话完成标记
session_cancelled         取消标记

验证方法:
· seq 字段单调递增 (从 0 开始)
· timestamp 单调非递减
· turnId 在整个流中保持一致
· sessionId 在整个流中保持一致
· 首事件必为 'session' (含 metadata)
· 尾事件必为 'done' 或 'error'
```

```
┌─────────────────────────────────────────────────────────────────┐
```

```
│                                                                  │
│  1. AI 输出的"结构验证"原则                                       │

│  3. Zod Schema 的"边界全覆盖"原则                                 │
│     → 每个 Schema: 正常 + 缺失必填 + 边界值 + 类型错误            │
│     → safeParse 代替 try-catch, 更精确                       │
```
